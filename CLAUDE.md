## Module Map

- `apps/web/` — Astro static site with one Preact island (`App.tsx`). Page title constant in `index.astro` is `'일본 노래 검색기'` (was `'노래방 검색기'`). Header is bilingual: Korean H1 `일본 노래 검색기` + English subtitle `Karaoke Search`. Search via MiniSearch (4-field index — `title_primary`, `title_ko`, `artist_primary`, `artist_ko`; no romaji). Reads `apps/web/public/data/songs.json`. Device-local favorites via localStorage key `karaoke-favorites:v1` (`useFavorites()` hook in `lib/favorites.ts`). Footer is restructured into 3 rows — a small uppercase `KARAOKE SEARCH` brand label, a bilingual data-accuracy disclaimer (`본 검색 결과는 참고용이며 실제 노래방 기기와 다를 수 있습니다 / Results are for reference; the actual karaoke machine may differ`), and a metadata row (`DB 업데이트 YYYY-MM-DD · MIT · 문제 보고 ↗ · GitHub ↗`) where `문제 보고` links to the GitHub issues tracker. The DB-update date is build-time pulled via `git log -1 --format=%cs`; metadata items are `white-space: nowrap` and the row wraps at bullet boundaries on slim viewports without orphaning single tokens. KR/EN bilingual UI strings (Japanese halves removed from chrome — artist content in `data/featured.ts` keeps kana). Theme system: light + dark both shipped via `prefers-color-scheme` (no manual toggle). `<html>` no longer carries `data-theme="dark"`; `<meta name="color-scheme" content="light dark">`. The `--rui-*` token family lives in `apps/web/src/styles/tokens.css` (imported via Astro frontmatter); `@font-face` declarations live in `apps/web/public/fonts/fonts.css` (linked via `<head>`); component-level CSS still lives in `index.astro`'s global `<style>` block. The old `--bg` / `--fg` / `--accent` / `--bg-elev` / `--bg-card` / `--fg-muted` / `--accent-fg` / `--border-strong` tokens are gone. Fonts: self-hosted Geist (display) + Inter (body) + Pretendard (Korean) under `apps/web/public/fonts/`, loaded via `@font-face` with `font-display: swap`. Japanese characters fall through to `Noto Sans JP` / system fonts (Pretendard JP intentionally not included). `--header-height` is `6rem` mobile / `6.38rem` desktop (set at `:root` and overridden in `@media (min-width: 720px) { :root }`; desktop value derived from rendered header height of 102.08px — was 6.5rem and left a ~1.93px stripe between header and sticky tab bar). Search-input wrap carries `padding-bottom: 0.5rem` so the focus-ring `box-shadow: 0 0 0 0.125rem var(--rui-fg)` doesn't get clipped under the sticky tab bar's opaque background. `App.tsx` also imports `VendorChips` (filters records by `karaoke_numbers.{tj,ky,joysound}` non-null) — vendor chips remain multi-select (outlined-pill `chip-group-vendor` modifier). Category filter is single-select with `전체` (All) chip as the default; type `CategoryFilter = Category | 'all'` exported from `CategoryChips.tsx` and reused across `App.tsx` and `lib/filter.ts` (`filterByCategory`); radio-group semantics with arrow-key keyboard nav. Chip-row layout: category and vendor chip groups + featured-artist strips all use single-line **horizontal scroll on mobile** (<720px) with hidden scrollbar (`scrollbar-width: none` + `::-webkit-scrollbar { display: none }`); each chip is non-breaking via `white-space: nowrap` + `flex-shrink: 0`. Tablet+ keeps DESIGN.md generous padding (14px 32px) and `flex-wrap: wrap` natural multi-row layout. Chip padding spec: mobile `6px 14px / 14px / 500`; tablet+ `14px 32px / 16px / 600`. Featured-artist strip: jpop 6 + vocaloid 5 + anime 6 (LiSA, Linked Horizon, 鈴木このみ, fripSide, EGOIST, ClariS). `<TabBar>` (Korean-only labels `검색` / `즐겨찾기` — no count badge; `favoriteCount` prop was dropped as dead code) sits at the top of `<main class="results">`, sticky beneath the header — underline-style tabs (not pill chrome). `App.tsx` holds an `activeTab` state of `'browse' | 'favorites'` (default `'browse'`, ephemeral — resets on reload, no URL hash, no localStorage tab key). `<FavoritesEmpty>` placeholder renders when on Favorites tab with zero favorites. `<NoResults>` renders just the title `검색 결과가 없습니다 / No matches`; the v2-promise hint paragraph + `.no-results-hint` CSS rule were dropped. `EmptyState` is now featured-artist-only — the favorites preview block was stripped. `matchesQuery` helper inside `App.tsx` does case-insensitive substring narrowing of favorites by query (over the same 4 search fields). Render-branch order in `App.tsx` is implemented via a typed `RenderMode` discriminator (`'error' | 'loading' | 'favorites-empty' | 'favorites' | 'browse-empty' | 'browse'`) and a `renderBody()` switch — replacing the prior nested ternary. The order is: error → loading → favorites-empty → favorites → browse-empty → browse. On Browse+empty during loading, `<EmptyState>` co-renders alongside the loading message to preserve the existing `cd54633` loading-mitigation behavior. The `'favorites'` and `'browse'` cases are a single fall-through (`case 'favorites': case 'browse':`) because their render output post-`results` computation is identical — candidate-set divergence happens upstream in the `useMemo`. The switch has a `default: const _exhaustive: never = mode; throw new Error(...)` arm so adding a 7th `RenderMode` member without a case fails `tsc --noEmit` at the assignment site. The favorites/browse list-render duplication is extracted into a `<ResultList>` component used by both branches. `App.tsx` receives a `songCount: number` prop derived at build time from `songs.json` length (passed by `index.astro` frontmatter). Result-card grid uses `align-items: stretch` with `flex: 1` cells so cards in a row align bottom-edges; `.result-numbers` row uses `margin-top: auto`. `App.tsx` `handleTabChange` resets `inputValue`/`query`/`categoryFilter`/`selectedVendors` and cancels any pending search-debounce timer on tab switch — favorites localStorage is preserved (the reset is ephemeral filter state only). `useFavorites()` return type no longer exposes the raw `favorites: Set<string>` member; consumers use `isFavorite(id)`, `toggle(id)`, and `orderedIds`. `.tab-bar` mobile sets `width: calc(100% + 2.5rem)` plus negative `margin-inline` to bleed edge-to-edge — `width: 100%` clamps to the parent content box so negative margins alone only shift, they don't extend; the 1rem gap below the tabs lives on the sibling selector `.tab-bar + .chip-group, .tab-bar + fieldset` so the tab `border-bottom` stays flush against the buttons. `.badge-number` pill padding is `6px 14px` (was `4px 10px` + mobile-only `min-height: 44px`) — slim profile matches `.badge-category`. Light-mode `--rui-pill-secondary-bg` is `#dfdfe2` (NOT a DESIGN.md §2 token; derived neutral chosen so pills stay JND-distinct from BOTH `--rui-surface` `#f4f4f4` (resting card) and `--rui-surface-hover` `#ececef` (hovered card)).
- `packages/schema/` — universal `SongRecord` type + Ajv validator. `Category = 'jpop' | 'vocaloid' | 'anime'` (3 values; the JSON Schema enum is derived from `CATEGORY_VALUES`). The `categories` field is `minItems: 1, maxItems: 1` enforcing 3-way mutual exclusivity (every record carries exactly one tag). The canonical `applyCategoryExclusivity(Set<Category>)` helper is exported here with priority `vocaloid > anime > jpop`, reused by the merger and by `scripts/ingest-anisong-pdf.py`. Compiled artifact at `packages/schema/dist/`. Both crawler and web depend on it.
- `packages/crawler/` — pluggable adapter pipeline. `Crawler` interface yields `SongRecord` directly (each adapter does its own normalize). Two adapters: `jpop-playlist-blog` (Tistory blog, ~21k records) and `tj-media-direct` (TJ catalog API, ~5.9k records). `mergeRecords` returns `MergeResult { records, conflicts }` — destructure at every call site. `http.ts` exposes `postForm(url, body)` (cache-bypass POST + form-encoded) and a `HostConfig` map keyed by hostname for per-host UA + rate-limit overrides. CLI at `packages/crawler/dist/cli.js` after build. The `tj-media-direct` adapter's filter chain is **cache-driven** (PR-2, 2026-04-29) — replacing the legacy loose-JP regex + ~170-entry Chinese-artist denylist with a 3-path confirmation: (1) per-record `nationalcode === 'JPN'` from `proEnrichmentMap` (sourced from `searchSong?strType=1` exact-`pro` match), (2) per-artist JPN tag from `artistNationalityMap` (sourced from a per-artist `searchSong?strType=2` vote tally), or (3) blog-whitelist rescue (defense-in-depth). PR-3 (`6dce50c`) trims the rescue whitelist by artist-script signal (rescue 1,165 → 141; ~30-50 incidental real-JP losses accepted). PR-4 (`f991f07`) splits multi-artist collab strings (`A & B`, `A feat. B`, `A,B`) before path-1 lookup so each component artist gets its own JPN tag, raising kept records from 5,953 → 6,022 (+69). Bootstrapping is a once-per-week sweep of `topAndHot100?strType=3` (JPOP charts) over the past 2 years to pre-seed confident-JPN artists for free. The same adapter also runs a per-record translit enrichment (PR-1) — for each surviving JPN record it pulls Korean transliterations (`sortTitleKo` → `title_ko`, `sortSongKo` → `artist_ko`). All cache fields share `apps/web/public/data/tj-search-cache.json` — tracked in git (~5.5 MB after PR-2 pre-seed + `0418820` expansion: `proEnrichmentMap` 4,447 entries, `artistNationalityMap` 26,230 entries with 1,531 confirmed JPN) so CI never pays the first-run cost. Per-entry TTL is 90 days; the Option-C chart bootstrap is gated on a 7-day freshness check. The `searchSong` HTTP helper strips ASCII apostrophes from `searchTxt` to work around a TJ server parser bug. On any enrichment HTTP error the affected record stays UNKNOWN/null — graceful degradation, no pipeline abort.

## Gotchas

- `--limit N` on the crawler caps **artist pages per adapter**, not output records. `BlogCrawler` round-robins jpop+vocaloid artists then appends mixed.
- `apps/web/public/data/songs.json` (~12 MB, 26k+ records, currently 26,480) is **tracked on `main`**. The deploy needs it baked into the build. Do NOT re-add it to `.gitignore`.
- Playwright e2e tests must use `await page.goto('')` (relative empty). `goto('/')` resolves the absolute path against the base, sending Playwright to the bare `github.io` domain (404).
- Schema runtime imports require `pnpm -r build` first — `@karaoke/schema` exports `dist/index.js`, not `src/index.ts`. CI builds with `pnpm -r build` before `pnpm start`.
- The blog adapter pulls artist names from the first `<blockquote>` in the post body. Posts without one (ranking lists, info posts) fall through; the index-parser filters known non-artist patterns (`랭킹|순위|차트|월간|연간|...`).
- Crawl workflow opens a PR labeled `crawl-output`; the GitHub repo setting **"Allow Actions to create and approve pull requests"** must be on, else the PR-create step fails.
- HTTP rate limit in `packages/crawler/src/http.ts` is 200ms base + ±50ms jitter (~4–6 req/sec) by default. Per-host overrides live in `HOST_CONFIG`: `www.tjmedia.com` is 500ms base + ±100ms jitter (no UA override — the catalog API is open). Adding hosts: extend the map; defaults apply otherwise.
- TJ-direct adapter (`packages/crawler/src/adapters/tj-media-direct/`) pulls the full catalog via one POST to `/legacy/api/newSongOfMonth` (`searchYm=200001`). Filter chain is the cache-backed 3-path confirmation in `parser.ts` (`shouldKeep`): (1) per-record `proEnrichmentMap[pro].nationalcode === 'JPN'`, (2) per-artist `artistNationalityMap[normalize(artist)].code === 'JPN'`, (3) blog-whitelist rescue (`forceIncludeTjNumbers`, set built from `apps/web/public/data/songs.json` at construction time). The legacy JP-regex + Chinese-artist denylist were removed in PR-2 (2026-04-29). Missing blog corpus file degrades silently to an empty rescue set with one `console.warn`.
- `mergeRecords` returns `{ records, conflicts }` — destructure at every call site. Two-tier match key: Tier A clusters by shared non-null `karaoke_numbers.{tj,ky,joysound}` (per-vendor union-find); Tier B falls back to normalized `(title_primary, artist_primary)` for residuals. Per-field ownership table picks the winner (TJ wins title/artist; blog wins Korean fields; vendor numbers union with `blog > namu > tj` tiebreak; categories set-union followed by `applyCategoryExclusivity` with priority `vocaloid > anime > jpop` — defense-in-depth: the JSON Schema enforces `maxItems: 1` on `categories` (combined with `minItems: 1` and the `Category` enum, every record carries exactly one tag from `{jpop, vocaloid, anime}`), and both the blog adapter and the Python PDF ingest call the canonical `applyCategoryExclusivity` helper from `@karaoke/schema` so pre-merger output is schema-conformant). The conflict aggregate is consumed by the crawl PR-body composition step in `.github/workflows/crawl.yml` when `total > 0`.
- Blog parser (`extractNumberCell`) returns `null` + warns on `<br>`-separated multi-code TJ/KY/JOY cells (instead of concatenating). Length-cap defensive guard: TJ/KY ≤6 digits, JOY ≤7. The merger's per-field ownership picks up the right value at merge time via Tier A on title+artist.
- Frontend favorites: backed by `localStorage` key `karaoke-favorites:v1` (versioned). The `useFavorites()` hook returns `{ favorites: Set, toggle, isFavorite, orderedIds }`. Stale ids (favorited then removed from corpus) are silently skipped at render. The Footer's git-log invocation has `cwd: repoRoot` resolved by `findRepoRoot()` in `Footer.astro`, which walks `import.meta.url` ancestors looking for a `.git` directory, then falls back to `process.cwd()`, then to `process.cwd()` with an unconditional `console.warn` if neither walk finds `.git`. The prior `../../../..` relative path was brittle: any future relocation of `Footer.astro` or change in Astro's build layout would silently land in the wrong directory. The `.git` walk is layout-independent.
- Favorites tab is mode-switch UI, not a filter chip. Active tab is in-memory only (resets on reload). Search box and category/vendor chips apply on both tabs. The render-branch order in `App.tsx` is driven by a typed `RenderMode` discriminator + `renderBody()` switch: error → loading → favorites-empty → favorites → browse-empty → browse — `loading` takes precedence over `FavoritesEmpty`, EXCEPT that on Browse+empty+loading the existing loading-mitigation co-renders `<EmptyState>` alongside the loading message (preserves commit `cd54633`'s behavior). The switch has an exhaustive `default: never` arm — a new `RenderMode` member without a corresponding case fails `tsc --noEmit`.
- `scripts/ingest-anisong-pdf.py` runs automatically as a post-step in `.github/workflows/crawl.yml` (after the JS crawler's atomic rename) so PDF section tags survive each weekly crawl. Writes are atomic: the script writes to a temp file then calls `os.replace()` to swap it in, preventing partial-write corruption. The cached text input at `.omc/anisong_utf8.txt` is **tracked in the repo** (~407 KB; whitelisted via `!.omc/anisong_utf8.txt` in `.gitignore`). The script is byte-idempotent: re-running on the unchanged cached text produces a 0-line diff against `apps/web/public/data/songs.json`. Regeneration of the cached text from a new source PDF is still a manual Windows-host step — `pdftotext -table -enc UTF-8 <pdf> .omc/anisong_utf8.txt` (Git-for-Windows ships xpdf's `pdftotext -table`; Linux poppler-utils does NOT support `-table`). Section tagging is data-driven via `_SECTION_DIVIDERS` in the script: `보컬로이드,` → vocaloid; `특촬물` → anime (tokusatsu collapses to anime per the 3-category schema). Unknown section names default to anime with a stderr warning. Regression tests at `scripts/test_ingest_anisong_pdf.py` cover the priority rule + anchor extraction + hangul transition + anime-row parse + anime→vocaloid section transition + unknown-section default-to-anime + full byte-idempotence round-trip on a 3-record synthetic corpus (17 cases via stdlib `unittest`).
- Font payload: ~1.5 MB total (Geist 47 KB + Inter 333 KB across 3 weights + Pretendard 1.07 MB across 4 weights). All `font-display: swap` so non-blocking; Pretendard JP intentionally NOT included since upstream only ships ~2 MB full files (no single-file subset). Japanese characters fall through to `Noto Sans JP` / system fonts.
- `--header-height` token must match actual rendered header content (h1 + subtitle + padding). Currently `6rem` mobile, `6.38rem` desktop via `@media (min-width: 720px) { :root }`. Editing the `.site-header` padding or `.site-title` / `.site-subtitle` font sizes requires re-deriving this value (rendered px ÷ 16) or the sticky `.tab-bar` will leave a visible background-color stripe between the header and tab strip at scroll. The desktop value was `6.5rem` until commit `532c0f6`, which dropped it to `6.38rem` (102.08px) — a sub-2px stripe was visible at scroll. Formula and rationale are pinned in a comment in `apps/web/src/styles/tokens.css`. TODO: migrate to a `ResizeObserver` on the rendered header so the value tracks typography changes automatically.
- Vite previously emitted 8 benign font-asset warnings during build (`/karaoke-search/fonts/*.woff2 didn't resolve at build time, it will remain unchanged to be resolved at runtime`). These are now **silenced** as a side-effect of the CSS extraction in commit `6b8763d`: `@font-face` declarations were moved to `apps/web/public/fonts/fonts.css` (a public-asset CSS file linked via `<head>`) so Vite no longer parses those `url()` paths at all. The woff2 files are still runtime-correct — Astro's public-asset copy places them in `dist/fonts/`.
- Schema validation gate: after the Python PDF ingest step in `crawl.yml`, a Node script `scripts/validate-songs-json.mjs` runs `validateSongRecord` from `@karaoke/schema` over every record in `apps/web/public/data/songs.json`. CI fails on any invalid record (first 5 sample failures are logged); after the sample, the script prints a "Failure summary" tally aggregated by 80-char message prefix so category-level signal is visible beyond just the first-5 sample IDs. This catches schema regressions the Python script doesn't track (e.g. a category mutation that violates `maxItems: 1`).
- Pinned GitHub Actions: all third-party actions in `crawl.yml` and `deploy.yml` are pinned by 40-char SHA with the original tag in a trailing comment (e.g. `actions/checkout@abc123...def456  # v4.2.2`). Upgrades require updating **both** the SHA and the trailing tag comment. Never update one without the other or the pin becomes misleading.
- E2E CI job: the deploy.yml E2E job runs against an in-CI `astro preview` server (NOT the deployed live site) using the `apps/web/dist` artifact uploaded by the build job. Currently set to `continue-on-error: true` with an inline review date: if the last 4 consecutive weekly runs are green by 2026-05-29, remove the flag and treat E2E as required; otherwise extend with a new review date. Do not remove the `continue-on-error` flag before that review.
- Theme is auto via `prefers-color-scheme` — no manual toggle UI. Adding one later is a single pill component writing to `<html data-theme>` and a couple of CSS-variable overrides.
- Category filter is single-select (radio-group), `전체` as default; vendor filter stays multi-select. Type `CategoryFilter = Category | 'all'` exported from `apps/web/src/components/CategoryChips.tsx` and reused across `App.tsx` and `lib/filter.ts`.
- Mobile-vs-desktop chip sizing intentionally diverges from DESIGN.md. DESIGN.md's generous 14px × 32px padding rule applies to primary CTAs; chips are secondary filter controls and use a tighter `6px 14px / 14px / 500` scale on <720px viewports. Tablet+ keeps the strict DESIGN.md spec (`14px 32px / 16px / 600`). Don't "fix" mobile back to the generous spec — it was deliberately shrunk because chips on a slim phone were overweight for a search-utility viewport.
- Horizontal-scroll chip rows (mobile <720px) hide their scrollbar entirely (`scrollbar-width: none` + `::-webkit-scrollbar { display: none }`); the touch-swipe + edge-bleed margins serve as the affordance. This is a deliberate UX choice — a visible scrollbar read as visual noise on a search utility — not an oversight; do not "restore" the scrollbar without UX sign-off.
- TJ-direct filter chain order matters: per-record (path 1) is the strongest signal (sourced from `searchSong?strType=1` exact-`pro` match — direct nationality tag from the API). Per-artist (path 2) is the **primary** confirmation path because per-record search has a ~33% miss rate (TJ search index doesn't reliably surface every title — observed in PR-1 pre-seed: 1,950 misses out of 5,961 fetches). Blog rescue (path 3) is the safety net for residual TJ-search index gaps; it remains a real DOM member of the rescue set, NOT dead code. The `searchSong` HTTP helper strips ASCII apostrophes (`'`) from `searchTxt` before sending — TJ's server has a parser bug returning `resultCode=04` on values containing them (observed on `pro=68988`, `pro=68992` IDOLiSH7 OST tracks). Stripping is a narrow, surgical workaround; do not "restore" the apostrophes via escape attempts (`\'`, `%27`, etc.) — all hit the same server bug.
- Footer DOM is a 3-row structure (`Footer.astro`): `<div class="site-footer-brand">` (uppercase `KARAOKE SEARCH`), `<p class="site-footer-disclaimer">` (two `<span>` halves KO/EN), and `<div class="site-footer-meta">` (metadata row of `<span class="site-footer-meta-item">` and `<a class="site-footer-meta-item site-footer-link">` items separated by `<span class="site-footer-meta-sep" aria-hidden="true">·</span>` bullets). The bullets are real DOM nodes, not inline punctuation — never assume the metadata row is plain text. Each meta item is `white-space: nowrap` so the row wraps at bullet boundaries on slim viewports without orphaning single tokens.
- TJ search cache (`apps/web/public/data/tj-search-cache.json`) is **tracked in git, NOT gitignored** — CI must not pay the first-run enrichment cost. Schema (PR-2): `{ version: 1, generatedAt, proEnrichmentMap: { [pro]: { nationalcode, sortTitleKo, sortSongKo, subTitle, publishdate, lastSeen } }, artistNationalityMap: { [normalize(artist)]: { code: 'JPN'|'KOR'|'ENG'|'AMBIGUOUS'|'UNKNOWN', votes: { JPN, KOR, ENG }, lastSeen } } }`. The cache is written atomically (`<file>.tmp` then rename), mirroring the `scripts/ingest-anisong-pdf.py` pattern. Per-`pro` and per-artist entries older than **90 days** are treated as missing by `isFresh()` / `isArtistNationalityFresh()` and re-fetched. The Option-C bootstrap (`bootstrapArtistMapFromCharts`) runs once per **7 days** — gated by `isBootstrapFresh(cache)` which checks `generatedAt` plus a non-empty `artistNationalityMap`. The merger's `koChain` is `['blog', 'namu', 'tj']` — TJ-search-translit fills Korean fields when blog and namu are silent. Pre-seeding the cache locally is the recommended cold-start path; CI builds today work with an empty seed but the per-artist scan would take ~1.4-2 hours on the first run. Forward-compat: any unrecognized top-level fields round-trip through an `extras` bag so future PRs can extend the schema without dropping unknown data.
- Mobile `.tab-bar` edge-bleed requires BOTH `width: calc(100% + 2.5rem)` AND negative `margin-inline` — `width: 100%` clamps to the parent content box so negative margins alone shift the box but don't extend it. The 1rem gap that previously sat on the tab-bar's `margin-bottom` was moved to the sibling selector `.tab-bar + .chip-group, .tab-bar + fieldset` so the underline `border-bottom` stays flush against the buttons. Don't restore the old `margin-bottom` rule.
- `--rui-pill-secondary-bg` (light mode) MUST stay distinct from BOTH `--rui-surface` (resting card BG) AND `--rui-surface-hover` (hovered card BG) on the JND threshold. Light mode currently uses `#dfdfe2` — stepped below both `#f4f4f4` (surface) and `#ececef` (surface-hover). Picking the same value as either token kills pill visibility on hover (PR `b28d4f0` then `0074c93` exist because of this collision class). This is NOT a DESIGN.md §2 token — it's a derived neutral; don't move it back into the canonical token doc.
- Tab switch (`activeTab` change in `App.tsx`) resets ephemeral filter state — `inputValue`, `query`, `categoryFilter`, `selectedVendors` — and cancels any pending search-debounce timer. Favorites localStorage (`karaoke-favorites:v1`) is preserved across switches. The reset behavior was added in `01994c4` because users were carrying stale filter chrome between Browse and Favorites tabs and seeing zero results inside their favorites set. Don't make the reset partial without UX sign-off.
- Manual crawl + push protocol: the JS crawler writes to `apps/web/public/data/songs.json.tmp` and the CI workflow performs the atomic rename via inline node (`node -e "...renameSync(...)"`). Running the same command locally on this Windows host triggers a sandbox block on the rename — the user must run the rename manually with elevated permission OR run the crawl directly inside a CI job. Don't attempt to bypass the sandbox.

## Git Conventions

- Conventional-commit prefixes: `docs:`, `chore:`, `feat:`, `fix:`, `refactor:`, `test:`, `ci:`, `ui:`.

## Orchestration (Project Rule)

- Main thread is orchestrator only. Delegate every work item to a specialized agent: `executor` (code), `code-reviewer` (review), `planner` (plans/specs), `document-specialist` (external docs), `verifier` (verification evidence), `explore` (codebase search), `git-master` (git ops).
- Never self-approve. Author and review are always separate agent passes.
- Trivial single-tool calls in service of orchestration (a single Read to route work, a Write of a memory entry) are OK from the main thread; everything else delegates.
